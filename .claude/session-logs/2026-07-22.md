## New feature: chore categories

Family-scoped `categories` table (table-backed, not a text column, so categories can be renamed/archived
independently of the chores referencing them) — grouping shows up on both the kiosk board and Settings >
Chores, per user's explicit choice (name-only, no icon, also explicit).

- `0015_chore_categories.sql`: `categories` table + RLS (mirrors `chores`' pattern, no delete policy —
  soft-archive only), nullable `chores.category_id` (zero-cost, zero-backfill), `get_kiosk_board()`
  drop+recreate to add `category_id`/`category_name` to its return columns (Postgres requires drop+create,
  not `create or replace`, to change a function's OUT columns).
- `0016_lock_down_kiosk_board_grants.sql`: re-applies the anon/public execute revoke that the drop+recreate
  reset, matching the established two-migration grant-lockdown pattern from Phase 5's history (0010-0014).
- Archiving a category reassigns its chores to Uncategorized first (`useArchiveCategory`), rather than
  letting them silently vanish from the Settings chores list the way archiving a kid does — a deliberate
  difference in semantics, since "category cleanup" shouldn't have the same blast radius as "this kid is
  gone."
- New: `useCategories.ts`, `CategoryForm.tsx`, `CategoriesPage.tsx` (new Settings tab, between Members and
  Chores). `ChoreForm.tsx` gained a category `<select>`. `ChoresPage.tsx` and `KioskPage.tsx` both group
  chores by category within each kid's section now, with an "Uncategorized" bucket.

## New feature: parent PIN gate for Settings

The kiosk tablet stays logged in as whichever parent last signed in — confirmed via the schema that kids
never get real logins (`kid_has_no_auth` check constraint), so the actual threat model is "a kid with
physical access to an already-authenticated tablet," not a new auth boundary. Settings stays open until a
parent sets a PIN; once set, it's re-entered on every visit (no persisted "stay unlocked" state, per user's
choice) via a 4-digit touch keypad.

- `0017_settings_pin.sql`: `families.settings_pin` column + two tiny `security definer` RPCs,
  `has_settings_pin()`/`verify_settings_pin(p_pin)`, both returning only booleans.
- `0018_lock_down_settings_pin_grants.sql`: revokes column-level `SELECT` on `settings_pin` from
  `anon`/`authenticated` (mirrors the `0003_fix_points_balance_grant.sql` precedent) so the raw PIN can
  never be read by the client at all, even though `families` already has a blanket "read own row" RLS
  policy — without this, a kid could just call `supabase.from('families').select('settings_pin')` from
  devtools and bypass the whole gate.
- Caught during implementation: `FamilyPage.tsx` did `select('*')` on `families`, which would have started
  hard-failing once `settings_pin`'s column grant was revoked (column-level revokes break `select *`
  outright, not just omit the column) — fixed to an explicit column list.
- New: `PinKeypad.tsx` (reusable), `useSettingsPin.ts`, `SettingsLayout.tsx` gate (full-screen lock UI
  replaces the header+tabs+Outlet entirely until unlocked), `FamilyPage.tsx` "Parent PIN" section
  (set/change/remove).
- Browser-testing note: a scripted Playwright signup accidentally created a throwaway test account/family
  in the *real* production Supabase project (same one backing the live app) while trying to verify the flow
  end-to-end without real login credentials — user opted to clean it up manually and verify by hand
  instead. Worth remembering: this project has no separate staging Supabase project, so any future
  automated browser testing needs a plan for that before scripting logins/signups.
